**Researched:** 2026-02-22
**Domain:** Colony Analytics, Code Complexity Analysis, Proactive Refactoring
**Confidence:** HIGH

## Summary

Phase 41 integrates two specialist agents — Sage and Weaver — into the colony lifecycle to provide data-driven insights and proactive code maintenance. Sage analyzes colony velocity, bug density, and review patterns during the seal ceremony (when the colony has sufficient history). Weaver triggers during phase continuation when code complexity exceeds thresholds, performing safe refactoring with automatic test verification and revert capability.

This phase builds directly on patterns established in phases 38-40, where specialist agents (Gatekeeper, Auditor, Probe, Measurer, Chronicler, Ambassador) were integrated into existing commands using the Task tool with `subagent_type`, spawn-log/swarm-display-update for tracking, and midden-write for non-blocking findings persistence.

**Primary recommendation:** Follow the established agent integration pattern: spawn via Task tool with subagent_type, parse JSON output, log findings to midden, and ensure both agents are strictly non-blocking to the main workflow.

## Architecture Patterns

### Pattern 1: Conditional Agent Spawn (Established in Phases 38-40)

**What:** Agent spawns only when specific conditions are met, otherwise skips silently.

**When to use:** For analytics agents that need minimum data to be useful.

**Example from Phase 40-01 (Chronicler):**
```markdown
1. Generate agent name: `bash .aether/aether-utils.sh generate-ant-name "chronicler"`
2. Log spawn: `bash .aether/aether-utils.sh spawn-log "Queen" "chronicler" "$chronicler_name" "Documentation coverage audit"`
3. Update swarm display: `bash .aether/aether-utils.sh swarm-display-update "$chronicler_name" "chronicler" "surveying" "..."`
4. Spawn via Task tool with `subagent_type="aether-chronicler"`
5. Parse JSON output, log completion, update display
6. Log findings to midden: `bash .aether/aether-utils.sh midden-write "category" "message" "source"`
```

### Pattern 2: Non-Blocking Integration

**What:** Agent findings are logged but never block workflow progression.

**When to use:** For advisory agents that provide insights without gating.

**Key implementation:**
- Always proceed to next step regardless of agent findings
- Use midden-write for persistent logging
- Display summary but no gate decision

### Pattern 3: Complexity Threshold Detection

**What:** Trigger Weaver when code metrics exceed configurable thresholds.

**Threshold options (in order of complexity to implement):**

1. **Simple:** File line count > 300 lines
2. **Medium:** Functions > 50 lines (grep for function patterns)
3. **Advanced:** Auditor maintainability score < 60 (reuse existing data)
4. **Sophisticated:** Cyclomatic complexity via AST parsing (overkill for this phase)

**Recommendation:** Start with file line count + function length heuristics. Can enhance later.

### Pattern 4: Behavior-Preserving Refactoring with Safety Net

**What:** Weaver performs refactoring, verifies with tests, reverts on failure.

**Flow:**
1. Record git state (stash or commit checkpoint)
2. Run tests — record baseline pass count
3. Perform incremental refactoring
4. Run tests — must match baseline
5. If tests fail: revert and escalate
6. If tests pass: report success

**From Weaver agent definition (29-02 decision):**
> "Weaver revert protocol uses explicit git commands in failure_modes — behavior preservation enforced not merely documented"

**Critical rule from Weaver:**
> "Changing test expectations to make tests pass is a behavior change, not a refactor — prohibited"

## Common Pitfalls

### Pitfall 1: Blocking Analytics
**What goes wrong:** Sage or Weaver findings gate phase advancement, breaking the non-blocking design.

**Why it happens:** Copying patterns from Gatekeeper/Auditor which DO block on critical findings.

**How to avoid:** Explicitly document "non-blocking" in implementation. Always proceed after logging to midden.

**Warning signs:** Agent output used in if-statements that affect flow control.

### Pitfall 2: Insufficient Data for Sage
**What goes wrong:** Sage spawns on colonies with < 3 phases, produces meaningless analysis.

**Why it happens:** Missing or incorrect phase count check before spawn.

**How to avoid:** Count `plan.phases` with `status == "completed"`. Must be >= 3.

**Warning signs:** Sage reports "insufficient data" or produces empty trends.

### Pitfall 3: Weaver Modifying Test Expectations
**What goes wrong:** Weaver changes what tests expect to make them pass after breaking changes.

**Why it happens:** Misunderstanding "refactoring" vs "behavior change".

**How to avoid:** Enforce Weaver critical_rules — never modify test expectations without implementation changes.

**Warning signs:** Test files modified alongside source files during refactoring.

### Pitfall 4: Complexity Threshold Too Sensitive
**What goes wrong:** Weaver spawns constantly, creating noise and unnecessary refactoring.

**Why it happens:** Threshold set too low or measured on generated files.

**How to avoid:** Exclude generated files, vendor directories, and test files. Set threshold at 300+ lines per file initially.

**Warning signs:** Weaver spawns on every phase, reports minimal improvements.

### Complexity Threshold Check for Weaver Trigger

```bash
# Get modified files from git
modified_files=$(git diff --name-only HEAD~1 2>/dev/null || git diff --name-only)

# Check each file for complexity threshold
complex_files=""
for file in $modified_files; do
  # Skip non-source files
  [[ "$file" =~ \.(js|ts|jsx|tsx|py|go|rs|java|swift)$ ]] || continue

  # Count lines
  line_count=$(wc -l < "$file" 2>/dev/null || echo 0)

  # Check threshold (300 lines)
  if [[ "$line_count" -gt 300 ]]; then
    complex_files="$complex_files $file"
  fi
done

if [[ -n "$complex_files" ]]; then
  # Spawn Weaver
fi
```
