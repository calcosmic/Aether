**Part A: Create handoff-build-error.template.md**

Create `.aether/templates/handoff-build-error.template.md`. This template covers the error recovery handoff when build workers fail (currently inline at Step 5.9 in build.md).

Use the Phase 21 template conventions:
- HTML comment header with template name, version (1.0), and fill instructions
- {{DOUBLE_BRACE}} placeholder syntax for markdown templates
- Content should be clear and actionable — this fires when things go wrong

Required placeholders (derived from the existing inline heredoc):
- {{PHASE_NUMBER}} — current phase number
- {{PHASE_NAME}} — current phase name
- {{BUILD_TIMESTAMP}} — ISO-8601 UTC timestamp
- {{FAILED_WORKERS}} — formatted list of failed workers (one per line: "- {ant_name}: {failure_summary}")
- {{GRAVE_MARKERS}} — formatted list of grave markers (one per line: "- {file}: {caution_level} caution")

Content structure (matching the existing inline heredoc but improved):
1. Header: "Colony Session — Build Errors"
2. Build Status section with phase info and timestamp
3. Failed Workers section with {{FAILED_WORKERS}} placeholder
4. Grave Markers section with {{GRAVE_MARKERS}} placeholder
5. Recovery Options: review failures, retry build, swarm fix, manual fix
6. Session Note: brief context for recovery

The voice should be clear and practical — this is a recovery document, not a ceremony. Direct, not emotional. The user needs to know what went wrong and how to fix it.

**Part B: Create handoff-build-success.template.md**

Create `.aether/templates/handoff-build-success.template.md`. This template covers the build success handoff (currently inline at Step 6.5 in build.md).

Required placeholders:
- {{GOAL}} — colony goal
- {{PHASE_NUMBER}} — current phase number
- {{PHASE_NAME}} — current phase name
- {{BUILD_STATUS}} — synthesis status
- {{BUILD_TIMESTAMP}} — ISO-8601 UTC timestamp
- {{BUILD_SUMMARY}} — synthesis summary text
- {{TASKS_COMPLETED}} — count of completed tasks
- {{TASKS_FAILED}} — count of failed tasks
- {{FILES_CREATED}} — count of created files
- {{FILES_MODIFIED}} — count of modified files
- {{SESSION_NOTE}} — conditional note based on build status

Content structure:
1. Header: "Colony Session — Build Complete"
2. Quick Resume section with /ant:continue guidance
3. State at Build Completion with goal, phase, status, timestamp
4. Build Summary with {{BUILD_SUMMARY}} placeholder
5. Tasks section with completed/failed counts
6. Files Changed section with created/modified counts
7. Next Steps: continue, flags, or pause options
8. Session Note: {{SESSION_NOTE}} placeholder

The voice should be informative and forward-looking — the build is done, here's what happened and what to do next. Professional, not ceremonial.
