3. **Wrap existing content in XML sections** (PWR-01 through PWR-05):

   `<execution_flow>` — Port the TDD Discipline section. Number each step:

   3. VERIFY RED: Run test, confirm failure with correct error
   4. GREEN: Write minimal code to pass
   5. VERIFY GREEN: Run test, confirm pass
   6. REFACTOR: Clean up while staying green
   7. REPEAT: Next test for next behavior
   Coverage target: 80%+ for new code.

   `<critical_rules>` — Port these rules from the OpenCode Builder:
   - **TDD Iron Law:** No production code without a failing test first.
   - **3-Fix Rule:** If 3+ attempted fixes fail on a bug, STOP and escalate with architectural concern — you may be misunderstanding the root cause.

   `<return_format>` — Port the JSON output format block (PWR-03). Remove the `spawns` field. Keep: ant_name, caste, task_id, status, summary, files_created, files_modified, tests_written, tdd block, blockers.

   `<failure_modes>` — Port directly from the OpenCode Builder's existing `<failure_modes>` section (PWR-05). Keep tiered severity (minor/major), the escalation format, and the 3-Fix Rule cross-reference. **Replace** any spawn references in escalation options with "escalate to the calling command" (e.g., replace "Spawn specialist" with "Request specialist via calling orchestrator").

   `<escalation>` — NEW section replacing Spawning Sub-Workers:

   ```
   <escalation>
   ## When to Escalate

   If you encounter a task 3x larger than expected or requiring genuinely different expertise, STOP and return status "blocked" with:

   - escalation_reason: why it exceeded scope
   - specialist_needed: what type of work is required

   The calling orchestrator (/ant:build, /ant:continue) handles re-routing.

   Do NOT attempt to spawn sub-workers — Claude Code subagents cannot spawn other subagents.
   </escalation>
   ```
