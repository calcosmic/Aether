## User Decisions (LOCKED)

### Spawn Triggers — Performance-Sensitive Phase Detection
Measurer spawns only for phases containing these keywords (case-insensitive):
- "performance"
- "optimize"
- "latency"
- "throughput"
- "benchmark"
- "speed"
- "memory"
- "cpu"
- "efficiency"

If phase name contains none of these keywords, Measurer skips silently.

### Integration Point
- Measurer runs after Step 5.5 (Watcher results) and before Step 5.6 (Chaos Ant)
- Insert as Step 5.5.1 between Watcher and Chaos
- Only runs if Watcher verification passed

### Agent Already Exists
- Measurer agent definition is at `.opencode/agents/aether-measurer.md`
- DO NOT create a new agent — integrate the existing one
- Spawn via Task tool with `subagent_type="aether-measurer"`

### Agent Constraints (from existing definition)
- Measurer is strictly read-only — never modifies code
- Uses Node.js built-in performance timing (no external dependencies)
- Logs findings to midden with category "performance" for trend analysis

### Baseline Storage
- Performance baselines stored in midden with file:line references
- Include: operation name, execution time, timestamp, phase context
- Future builds can compare against these baselines

Insert new Step 5.5.1 "Measurer Performance Agent (Conditional)" after Step 5.5 (Process Watcher Results) and before Step 5.6 (Spawn Chaos Ant).

The new step must:

1. Check if phase is performance-sensitive:
   - Extract phase name from COLONY_STATE.json (already loaded in build)
   - Check for keywords: "performance", "optimize", "latency", "throughput", "benchmark", "speed", "memory", "cpu", "efficiency"
   - Case-insensitive match
   - If no keywords match: Skip Measurer silently, continue to Chaos Ant
   - Display: "📊🐜 Measurer: Phase not performance-sensitive — skipping baseline measurement"

2. Check Watcher results:
   - Only spawn Measurer if Watcher verification passed
   - If Watcher failed: Skip Measurer (unreliable performance data on broken code)
   - Display: "📊🐜 Measurer: Watcher verification failed — skipping performance measurement"

3. Generate Measurer name and dispatch:
   - Run: `bash .aether/aether-utils.sh generate-ant-name "measurer"`
   - Store as `{measurer_name}`
   - Log spawn: `bash .aether/aether-utils.sh spawn-log "Queen" "measurer" "{measurer_name}" "Performance baseline measurement"`
   - Update swarm display for visual mode
   - Display: "📊🐜 Measurer {measurer_name} spawning — Establishing performance baselines for {phase_name}..."

4. Get files to measure:
   - Use files_created and files_modified from builder results (already collected in synthesis)
   - Filter for source files (not test files, not config files)
   - Pass to Measurer as context

5. Spawn Measurer using Task tool with `subagent_type="aether-measurer"`:
   - **CRITICAL:** The agent already exists at `.opencode/agents/aether-measurer.md`
   - Include fallback: If "Agent type not found", use general-purpose and inject Measurer role
   - Pass mission context with:
     - Phase name and number
     - List of source files to measure
     - Performance keywords that triggered spawn
   - Request JSON output matching Measurer schema (see agent definition)

6. Parse Measurer JSON output:
   - Extract: `metrics`, `bottlenecks_identified`, `recommendations`
   - Log completion: `bash .aether/aether-utils.sh spawn-complete "{measurer_name}" "completed" "Baselines established, bottlenecks identified"`

7. Log findings to midden:
   - For each baseline: `bash .aether/aether-utils.sh midden-write "performance" "Baseline: {metric} = {value} at {file}:{line}" "measurer"`
   - For each bottleneck: `bash .aether/aether-utils.sh midden-write "performance" "Bottleneck: {description} ({severity}) at {location}" "measurer"`
   - For each recommendation: `bash .aether/aether-utils.sh midden-write "performance" "Recommendation: {recommendation}" "measurer"`

8. Display summary and continue:
   - Display: "📊🐜 Measurer complete — Baselines and bottlenecks logged to midden"
   - Continue to Step 5.6 (Chaos Ant)

9. Update build synthesis to include Measurer results:
   - Add to synthesis JSON: `performance: { from Measurer output }`
   - Include in BUILD SUMMARY display if Measurer ran
   - Show: `📊 Measurer: baselines established, bottlenecks identified`
