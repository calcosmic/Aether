Implement file-based worker heartbeat liveness detection (PLAT-03).

Output: New heartbeat monitor goroutine, heartbeat prompt instruction in worker briefs, and cleanup integration.
</objective>

  <action>
Per D-01 and D-03, create `cmd/heartbeat_monitor.go` with:

1. **HeartbeatFile struct:**
```go
type HeartbeatFile struct {
    WorkerID  string `json:"worker_id"`
    Caste     string `json:"caste"`
    Timestamp string `json:"timestamp"` // RFC3339
    Phase     int    `json:"phase"`
}
```

2. **Constants:**
- `heartbeatFilePrefix = "heartbeat-"` -- prefix for heartbeat files
- `heartbeatScanInterval = 15 * time.Second` -- how often the goroutine scans
- `heartbeatStaleWarningThreshold = 90 * time.Second` -- 3x the ~30s write interval before warning (per RESEARCH open question recommendation)
- `heartbeatCleanupPattern = "heartbeat-*.json"` -- glob for cleanup

3. **StartHeartbeatMonitor(ctx context.Context, dataDir string) context.CancelFunc:**
- Creates a goroutine with a `time.NewTicker(heartbeatScanInterval)`
- On each tick, calls `scanHeartbeatFiles(dataDir)`
- Returns a cancel function that cancels the context and stops the goroutine
- Uses `emitVisualProgress()` to report stale workers (e.g., "Heartbeat stale for worker Hammer-23 (last seen 2m30s ago)")

4. **scanHeartbeatFiles(dataDir string):**
- Reads `.aether/data/` directory using `os.ReadDir(dataDir)`
- For each file matching `heartbeat-*.json` prefix: read JSON into `HeartbeatFile`, parse timestamp, compare with `time.Since()`
- If stale (time.Since > heartbeatStaleWarningThreshold): emit visual warning with worker ID and time since last heartbeat
- Skip malformed JSON files silently (do not error)
- Use `filepath.Base()` on filenames to prevent path traversal (security per STRIDE threat T-92-01)

5. **cleanupHeartbeatFiles(dataDir, workerID string):**
- Remove `.aether/data/heartbeat-{workerID}.json` if it exists
- Called from `UntrackProcess` integration path and build-completion cleanup
- Returns error only for unexpected failures (os.IsNotExist is fine)

6. **cleanupAllHeartbeatFiles(dataDir string):**
- Remove all `heartbeat-*.json` files in dataDir
- Called at build completion to clean up all heartbeat artifacts

Create `cmd/heartbeat_monitor_test.go` with unit tests for all the above behaviors. Use temp directories for file I/O. Mock `emitVisualProgress` by capturing output via the existing test helper patterns (see `codex_build_test.go` for stdout capture pattern with `saveGlobals`).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Worker -> heartbeat file | Untrusted LLM agent writes heartbeat file to .aether/data/ |
| Go runtime <- heartbeat file | Go runtime reads and parses untrusted JSON from workers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-92-01 | Tampering | scanHeartbeatFiles | mitigate | Use filepath.Base() on filenames; validate JSON structure before processing; skip malformed files silently |
| T-92-02 | Tampering | HeartbeatFile path | mitigate | Worker ID in heartbeat filename is derived from deterministic assignment, not user input |
| T-92-03 | Denial of Service | Heartbeat monitor goroutine | accept | Goroutine has minimal overhead (one directory scan every 15s); accepts risk of minimal CPU usage |
</threat_model>
