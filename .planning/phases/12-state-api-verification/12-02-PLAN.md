**Part A: verify-claims subcommand**

Add a `_verify_claims` function to aether-utils.sh (place it near the existing state-write/state-checkpoint area, roughly after the state management subcommands). The function takes three arguments:
- `$1` -- path to builder claims JSON file (last-build-claims.json)
- `$2` -- path to watcher claims JSON file (or inline JSON)
- `$3` -- test exit code from the verification loop

The function performs two checks:

**Check 1: Builder-claimed files exist (LOCKED: hard block on missing)**
Read `files_created` and `files_modified` arrays from the builder claims JSON. For each path, check `[[ -f "$path" ]]`. Missing file = add to mismatches array and set `blocked=true`.

**Check 2: Test exit code vs Watcher verification_passed (hard block -- per research recommendation)**
Read `verification_passed` from watcher claims. If test exit code != 0 but watcher claims `verification_passed: true`, this is fabrication. Add to mismatches and set `blocked=true`.

Output JSON result:
```json
{
  "ok": true,
  "result": {
    "verification_status": "passed" | "blocked",
    "checks_run": 2,
    "mismatches": [],
    "blocked": false,
    "summary": "Verification passed" | "Worker claimed 3 files modified, but X doesn't exist. Blocked."
  }
}
```

The summary must be a plain one-liner per the locked decision on failure communication.

Add the case entry: `verify-claims) _verify_claims "$@" ;;`

Edit `.aether/docs/command-playbooks/continue-verify.md` to add a new step **between the current verification report display (around line 300 "Continue to Step 1.6") and the Spawn Enforcement Gate (Step 1.6 in continue-gates.md)**.

Add this section after the "If READY" block and before "Continue to Step 1.6":

```markdown
### Step 1.5.3: Verify Worker Claims (MANDATORY)

Cross-reference worker claims against reality. This step catches fabricated success claims.

**Always runs. No skip flag.**

1. Check if `.aether/data/last-build-claims.json` exists.
   - If not found: Display "No builder claims file found -- skipping file verification" and continue. (This handles first-time runs and manual builds.)

2. Capture the test exit code from Phase 4 of the verification loop (store it in a variable during Phase 4 execution above).

3. Run verification:
   Run using the Bash tool with description "Verifying worker claims...":
   ```bash
   bash .aether/aether-utils.sh verify-claims ".aether/data/last-build-claims.json" "<watcher_json_or_path>" "<test_exit_code>"
   ```

   For the watcher JSON: use the Watcher output from the most recent build (if available in COLONY_STATE.json events or build synthesis). If no Watcher output is available, pass `'{"verification_passed":true}'` as default (conservative -- only test exit code mismatch can trigger).

4. Parse the result:

   **If verification_status is "passed":**
   ```
   Verification passed
   ```
   Continue to Step 1.6.

   **If verification_status is "blocked" AND this is the first attempt (retry_count == 0):**
   Display each mismatch as a plain one-liner:
   ```
   Verification issue: Worker claimed src/api/auth.ts was created, but file does not exist.
   Retrying build for current phase...
   ```

   Log a flag for each mismatch:
   Run using the Bash tool with description "Flagging verification mismatch...":
   ```bash
   bash .aether/aether-utils.sh flag-create "Verification mismatch: <summary>" --type blocker
   ```

   **Auto-retry once** (locked decision):
   - Re-run `/ant:build <current_phase>` for the current phase
   - After retry build completes, re-run the verification loop from Phase 1
   - If verify-claims passes on retry: Display "Retrying... passed on retry" and continue
   - If verify-claims still blocked on retry: proceed to hard stop below

   **If verification_status is "blocked" AND retry_count >= 1 (retry already attempted):**
   ```
   Verification failed after retry.
   <plain summary of each mismatch>
   Phase will NOT advance. Fix the issues and run /ant:continue again.
   ```

   **CRITICAL:** Do NOT proceed to Step 1.6. Do NOT advance the phase. The verification failure is a hard block just like a test failure.
```

- The retry mechanism reuses the existing `/ant:build` command, which is already available in the continue flow context
- Flag creation uses the existing `flag-create` subcommand (already available)
- The "always show verification status" requirement (locked decision) means BOTH success and failure display a message
