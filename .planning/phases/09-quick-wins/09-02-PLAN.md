**REL-05: state-write subcommand (aether-utils.sh)**

Add a new `state-write` case in the main case statement of `aether-utils.sh`. Place it near the `state-checkpoint` subcommand (added in Plan 01). The subcommand:

1. Reads JSON content from `$1` argument, or from stdin if no argument provided:
   ```
   sw_content="${1:-}"
   if [[ -z "$sw_content" ]]; then
     sw_content=$(cat)
   fi
   ```

2. Validates the content is valid JSON using `jq -e .`. If invalid, call `json_err "$E_JSON_INVALID" "state-write received invalid JSON"`.

3. Acquires a lock on COLONY_STATE.json:
   ```
   acquire_lock "$sw_state_file" || json_err "$E_LOCK_FAILED" "Failed to acquire lock on COLONY_STATE.json"
   ```
   Use the standard `LOCK_DIR` (which is `$AETHER_ROOT/.aether/locks`), NOT the hub-level lock dir.

4. Creates a backup before writing:
   ```
   if [[ -f "$sw_state_file" ]]; then
     create_backup "$sw_state_file" 2>/dev/null || true
   fi
   ```

5. Writes atomically using `atomic_write "$sw_state_file" "$sw_content"`. On failure, release lock and call `json_err`.

6. Releases lock and returns `json_ok '{"written":true}'`.

7. Important: wrap all error paths so the lock is always released. Use this pattern:
   ```
   atomic_write "$sw_state_file" "$sw_content" || {
     release_lock 2>/dev/null || true
     json_err "$E_INTERNAL" "Failed to write COLONY_STATE.json"
   }
   release_lock 2>/dev/null || true
   ```

**REL-06: Budget trimming notification (aether-utils.sh)**

In `.aether/aether-utils.sh`, the colony-prime case, after the budget enforcement section (after line 8494, before the JSON escaping at line 8497):

1. Initialize trimmed notice variables:
   ```
   cp_trimmed_notice=""
   cp_trimmed_high_priority=false
   ```

2. If `cp_budget_trimmed_list` is non-empty, build the notice:
   - Check if high-priority items were trimmed: `key-decisions` or `pheromone-signals` in the trimmed list
   - Format the trimmed sections list (replace commas with ", " for readability)
   - For high-priority: use `[!trimmed]` marker and append warning about missing key decisions or redirect signals
   - For normal: use `[trimmed]` marker with the section list
   - Emit the notice to stderr using `echo "..." >&2`

   Use `[trimmed]` for normal and `[!trimmed]` for high-priority (per Claude's Discretion — square brackets are terminal-safe and grep-friendly, distinct from recovery messages that use `⚠`).

3. Add `trimmed_notice` and `trimmed_high_priority` to the JSON output. In the `jq -n` call at line 8502-8520:
   - Add `--arg trimmed_notice "$cp_trimmed_notice"` parameter
   - Add `--argjson trimmed_high_priority "${cp_trimmed_high_priority:-false}"` parameter
   - Add `trimmed_notice: $trimmed_notice` and `trimmed_high_priority: $trimmed_high_priority` to the JSON object

The exact code structure (see RESEARCH.md Pattern 6 for reference):

```bash
cp_trimmed_notice=""
cp_trimmed_high_priority=false

if [[ -n "$cp_budget_trimmed_list" ]]; then
  cp_trimmed_sections=$(echo "$cp_budget_trimmed_list" | tr ',' ', ')

  if [[ "$cp_budget_trimmed_list" == *"key-decisions"* ]] || \
     [[ "$cp_budget_trimmed_list" == *"pheromone-signals"* ]]; then
    cp_trimmed_high_priority=true
    cp_trimmed_notice="[!trimmed] Context exceeded ${cp_max_chars}-char budget. Dropped: ${cp_trimmed_sections}. HIGH-PRIORITY items were trimmed — key decisions or redirect signals may be missing."
    echo "[!trimmed] Colony context exceeded budget. High-priority sections dropped: $cp_trimmed_sections" >&2
  else
    cp_trimmed_notice="[trimmed] Context exceeded ${cp_max_chars}-char budget. Dropped: ${cp_trimmed_sections}."
    echo "[trimmed] Colony context exceeded budget. Dropped: $cp_trimmed_sections" >&2
  fi
fi
```

Per user decisions:
- Show what was trimmed AND why (which sections dropped, what budget was exceeded) -- done via the section names and budget number
- Show the notice every time trimming occurs -- done by emitting on every call where trimming happens
- Escalate for high-priority items -- done via `[!trimmed]` vs `[trimmed]`
- Use a different marker than `⚠` -- done via `[trimmed]` brackets
