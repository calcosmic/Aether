       Export interface `LifecycleOptions extends DispatchOptions`: `{ phase?: number }` (default: 1)

       Export async function `runLifecycle(opts: LifecycleOptions): Promise<{ success: boolean; error?: string }>`:

       **Step 1 -- Plan:**
       - Call `callGoJSON<PlanCompletion>(opts, ["plan", "--plan-only", "--depth", "fast"])`
       - Extract the plan manifest from the result. Go plan --plan-only returns JSON with `plan_manifest` containing phases, dispatches, etc.
       - Build plan completion: `{ plan_manifest: planManifest, dispatches: planManifest.dispatches.map(d => ({ ...d, status: "completed", summary: "Planning dispatch completed by TS host" })) }` -- all planning dispatches are marked completed since TS host is orchestrating.
       - Write completion file using `writeCompletionFile(os.tmpdir(), "plan-completion.json", { result: planCompletion })`
       - Call `callGoJSON(opts, ["plan-finalize", "--completion-file", completionFilePath])`
       - Log to stderr: "Plan finalized successfully"

       **Step 2 -- Build:**
       - Call `callGoJSON(opts, ["build", String(opts.phase ?? 1), "--plan-only"])` to get build manifest
       - The Go output for build --plan-only nests the manifest under `result.dispatch_manifest` (verify by reading codex_build.go plan-only output path). Extract the dispatches from the correct nested path.
       - Call `dispatchWorkers(opts, manifest.dispatches)` to dispatch all workers with spawn-log/complete
       - Convert results using `toWorkerResults(manifest.dispatches, workerResults)`
       - Build completion: `{ dispatch_manifest: manifest, dispatches: workerResultsAsWorkerResult }`
       - Write completion file using `writeCompletionFile(os.tmpdir(), "build-completion.json", { result: buildCompletion })`
       - Call `callGoJSON(opts, ["build-finalize", String(opts.phase ?? 1), "--completion-file", completionFilePath])`
       - Log to stderr: "Build finalized successfully"

       **Step 3 -- Continue:**
       - Call `callGoJSON(opts, ["continue", "--plan-only"])` to get continue manifest
       - Build continue completion: `{ continue_manifest: continueManifest, dispatches: [] }` (empty dispatches since TS host is not dispatching review workers for this prototype)
       - Write completion file using `writeCompletionFile(os.tmpdir(), "continue-completion.json", { result: continueCompletion })`
       - Call `callGoJSON(opts, ["continue-finalize", "--completion-file", completionFilePath])`
       - Check if result contains `blocked: true` -- if so, log "Continue blocked by gates" but still return success (gates are informational)
       - Log to stderr: "Continue finalized successfully"

       **Error handling:**
       - Wrap entire lifecycle in try/catch
       - On error, log to stderr with step context ("Failed at step N: ...")
       - Return `{ success: false, error: message }`
       - Per HOST-07: if the lifecycle cannot complete, the error message serves as the documented blocker
