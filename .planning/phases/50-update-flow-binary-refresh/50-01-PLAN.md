# Plan 50-01: Add binary download to the update flow

## Objective

When a repo updates and gets new YAML files that reference `aether <cmd>`, the update
flow should check if the Go binary exists on PATH. If not, download and install it from
the GitHub Release produced by goreleaser (phase 48).

## Context

- Phase 48 (goreleaser) produces platform tarballs on GitHub Releases
- Phase 49 (binary-downloader) created `bin/lib/binary-downloader.js` which handles
  platform detection, SHA-256 verification, and atomic install to `~/.aether/bin/aether`
- Currently `downloadBinary` is only called during `performGlobalInstall` (npm postinstall)
- The `aether update` command syncs files from hub to repo but never touches the binary

## Tasks

### Task 1: Add binary refresh to the update command

**What:** In `bin/cli.js`, the `update` command handler calls `updateRepo()` which syncs
system files, commands, and agents from hub. After the file sync completes successfully,
add a step that:

1. Checks if `~/.aether/bin/aether` exists
2. If missing, calls `downloadBinary(VERSION)` from `bin/lib/binary-downloader.js`
3. If present, checks the installed version against `VERSION` and downloads if newer
4. Wraps the whole thing in try/catch so failure is non-blocking (logs warning, continues)

**Where:** `bin/cli.js` — in both the single-repo update path (line ~1620) and the
`--all` update path (line ~1500). Extract into a helper function like
`refreshBinary(version)` to avoid duplication.

**Test:** Add tests to `tests/unit/binary-downloader.test.js` verifying:
- `refreshBinary` is called during update when binary is missing
- `refreshBinary` is skipped when binary exists and version matches
- Update completes successfully even if `refreshBinary` throws

### Task 2: Non-blocking guarantee (UPD-02)

**What:** Verify and enforce that a binary download failure never blocks the update
flow. The file sync (commands, agents, system files) must always complete regardless
of binary download outcome.

**Where:** The `refreshBinary` call must be after the file sync, in a try/catch that
logs the failure but never rethrows. Add a clear comment: `// Binary refresh is
non-blocking — update always completes`.

**Test:** Test that `updateRepo` succeeds even when `downloadBinary` rejects.

## Key Files

- `bin/cli.js` — update command handler, add `refreshBinary()` helper
- `bin/lib/binary-downloader.js` — existing downloadBinary module
- `tests/unit/binary-downloader.test.js` — add update-flow integration tests

## Success Criteria

- [ ] Running `aether update` downloads binary when missing or outdated
- [ ] Binary download failure does not block file sync
- [ ] Tests pass for both success and failure paths
